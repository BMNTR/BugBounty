# Mobile Reverse Engineering Skill

This skill provides methodologies for tearing apart compiled Android applications, understanding their logic, and modifying them.

## Workflow

### 1. Decompilation & Static Analysis
- **Resource Extraction:** Use `apktool d target.apk` to decode the `AndroidManifest.xml` to human-readable format and extract resources (`res/` folder) and Smali code.
- **Java Decompilation:** Use `jadx-gui target.apk` (or `jadx -d out target.apk`) to decompile DEX files into readable Java source code. This is the primary method for code review.
- **Identify Protections:** Look for evidence of ProGuard/R8 (obfuscated class names like `a.b.c`), DexGuard (encrypted strings), or packers (e.g., Qihoo, Secneo).

### 2. Code Review & Mapping
- Analyze `AndroidManifest.xml` to map out the application's attack surface (Exported Activities, Services, Receivers, Providers).
- Search the decompiled Java code for:
  - Cryptographic implementations (e.g., `Cipher.getInstance()`, `SecretKeySpec`).
  - Network requests (e.g., `OkHttpClient`, `HttpURLConnection`, `Retrofit`). Look for hardcoded endpoints or API configurations.
  - Authentication logic (e.g., `verifyPin`, `checkPassword`, `isRooted`).

### 3. Modifying & Patching (Smali Injection)
If you need to bypass a local check (e.g., forcing a `checkPremium()` function to return true):
- Locate the target function in `jadx`. Note the class name and method name.
- Open the corresponding `.smali` file generated by `apktool`.
- Modify the Dalvik bytecode (Smali).
  - Example: Change a condition (`if-eqz`) or force a return value (`const/4 v0, 0x1` then `return v0`).
- **Recompile:** `apktool b target_folder -o patched.apk`
- **Sign:** Sign the patched APK using `apksigner` and `keytool` before installing it on a device.

### 4. Native Library Analysis (JNI / .so files)
- If the application relies on native C/C++ libraries (found in `lib/`), use tools like `Ghidra` or `IDA Pro` to decompile the `.so` files.
- Look for hardcoded secrets, custom encryption routines, or anti-debugging mechanisms hidden in native code, which is harder to reverse engineer than Java.
